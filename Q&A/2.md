**Q11: You're profiling a GEMM kernel and Nsight Compute shows "SM: Pipe FMA Cycles Active = 14.81%" but "SM: Inst Executed Pipe Lsu = 76.63%". The overview shows 76% compute throughput. Your manager is excited about the high number, but you're concerned. Explain what's actually happening here and what these metrics really mean for your kernel's performance.**

**A:** This is a classic profiler interpretation trap that catches even experienced developers. The manager is looking at the wrong number.

**What's actually happening:**

The "76.63% compute throughput" in the overview is misleading because it's not measuring FP32 computation—it's measuring **Load/Store Unit** (LSU) pipeline utilization. The profiler shows the maximum across all execution pipelines, and here LSU dominates.

Breaking down what each metric means:

**SM: Inst Executed Pipe Lsu = 76.63%**
- This measures how busy the Load/Store pipeline is
- It handles loads, stores, atomics, and reductions to L1TEX
- High utilization here means we're spending most cycles moving data around
- This is **not** compute—it's memory operations

**SM: Pipe FMA Cycles Active = 14.81%**
- This is the real compute metric we care about
- Measures what fraction of SM cycles the FP32 FMA units are actually doing work
- Only ~15% means we're barely using our arithmetic hardware
- For a good GEMM, we'd want 60-80%+ here

**SM: Mio Inst Issued = 40.08%**
- Memory Input/Output instruction issue rate
- Together with LSU at 76%, confirms memory operations dominate

**The fundamental problem:**

We're memory-bound, not compute-bound. The instruction mix is roughly 50/50 between memory ops and everything else, but only a tiny fraction is actual FP32 arithmetic. The kernel spends most of its time orchestrating data movement rather than computing.

**Why the overview number is misleading:**

The "compute throughput" label in the overview section considers *all* SM instructions—ALU, FMA, SFU, LSU combined. It picks the maximum utilization across these pipelines. Since LSU is at 76%, that's what gets displayed, but it has nothing to do with actual floating-point computation.

**What to do about it:**

1. **Increase arithmetic intensity**: Do more work per byte loaded
   - Each thread should compute multiple output elements (register tiling)
   - Reuse data from shared memory across many operations

2. **Reduce memory traffic**:
   - Use shared memory tiling to avoid redundant GMEM accesses
   - Vectorize loads (float4) to reduce instruction count
   - Ensure coalesced access patterns

3. **Profile target**: For a GEMM kernel, aim for:
   - SM: Pipe FMA Cycles Active: 60-80%
   - SM: Inst Executed Pipe Lsu: 15-25%
   - This inverts the current ratio

The key insight for the manager: "We're achieving 76% utilization of our data movement hardware, but only 15% utilization of our actual compute hardware. We're paying for FP32 performance but mostly moving data around. We need to restructure the kernel to do more math per byte loaded."

**Q12: You're implementing a custom fused kernel and need to decide between using vectorized loads (float4) or sticking with scalar loads. You've heard "vectorization is always better," but when you try float4 loads in your kernel, performance actually gets worse. The profiler shows your SMEM bank conflict rate went from 1-way to 5-way. What's going on, and how do you think through the vectorization decision systematically?**

**A:** This is a perfect example of how "best practices" can backfire without understanding the underlying constraints. Vectorization isn't free, and it comes with layout requirements that can create problems elsewhere.

**What happened - the bank conflict story:**

With scalar loads, you were probably doing something like:
```cuda
for (int i = 0; i < TILE_SIZE; i++) {
    reg[i] = sharedA[row * STRIDE + i];
}
```

When you "optimized" to float4:
```cuda
for (int i = 0; i < TILE_SIZE; i += 4) {
    float4 tmp = reinterpret_cast<float4*>(&sharedA[row * STRIDE + i])[0];
    reg[i] = tmp.x; reg[i+1] = tmp.y; 
    reg[i+2] = tmp.z; reg[i+3] = tmp.w;
}
```

**The problem**: float4 loads require **16-byte alignment**. This means the data must be laid out so that consecutive elements you're loading span exactly 4 consecutive banks in shared memory.

Let's say your STRIDE = 128 and threads in a half-warp have different row values but are loading from column positions that follow a pattern like:
- Thread 0: loads from columns [0, 1, 2, 3]
- Thread 1: loads from columns [8, 9, 10, 11]  
- Thread 2: loads from columns [16, 17, 18, 19]
- ...

Bank calculation: `bank = (word_index) % 32`

For thread 0's float4 starting at column 0:
- Words 0, 1, 2, 3 → banks {0, 1, 2, 3}

For thread 1's float4 starting at column 8:
- Words 8, 9, 10, 11 → banks {8, 9, 10, 11}

This looks fine so far. But here's where it breaks: if your access pattern has threads within a warp accessing with a stride that causes bank aliasing.

**Example of the actual problem:**

Suppose COLS_PER_THREAD = 8, and threads are organized such that:
```
tx = threadIdx.x % 16  // half-warp
column_offset = tx * 8
```

When loading with float4:
```
bank_start = (i * STRIDE + tx * 8) % 32
```

If STRIDE = 128 and 128 % 32 = 0, then:
```
bank_start = (tx * 8) % 32
```

For threads 0-15 in a half-warp:
- Thread 0: (0 * 8) % 32 = 0 → banks {0,1,2,3}
- Thread 1: (1 * 8) % 32 = 8 → banks {8,9,10,11}
- Thread 2: (2 * 8) % 32 = 16 → banks {16,17,18,19}
- Thread 3: (3 * 8) % 32 = 24 → banks {24,25,26,27}
- Thread 4: (4 * 8) % 32 = 0 → banks {0,1,2,3}  ← CONFLICT with thread 0!

Threads 0 and 4 both need banks {0,1,2,3}, threads 1 and 5 both need {8,9,10,11}, etc. This creates 4-way conflicts, and if you have a full warp, it can go to 5-way or worse.

**With scalar loads**, each thread only accessed one bank at a time, and even if there were conflicts, they were 1-way (each access was independent).

**How to think through vectorization systematically:**

**Step 1: Check access pattern alignment**
- Will the base addresses be 16-byte aligned?
- Are consecutive elements you're loading actually contiguous in memory?

**Step 2: Bank conflict analysis**
- Map out which banks each thread in a warp will access
- Look for patterns where multiple threads hit the same bank
- Consider both the base address and the stride

**Step 3: Cost-benefit calculation**
- Vectorization benefit: Reduces instruction count by ~4× (one LDS.128 vs four LDS.32)
- Vectorization cost: Bank conflicts can serialize access by 4-5×
- Net result: 4× instruction reduction with 5× serialization = performance loss

**Step 4: Solutions if you want vectorization:**

**Option A: Padding**
```cuda
constexpr int STRIDE = (TILE_SIZE % 32 == 0) ? (TILE_SIZE + 4) : TILE_SIZE;
```
Add 4 floats of padding to break the stride's divisibility by 32. Now:
```
bank_start = (i * 132 + tx * 8) % 32
```
The 132 (padded stride) is no longer divisible by 32, so the row index `i` influences which bank you hit, distributing accesses across banks.

**Option B: Transpose + vector load**
- If you're loading down a column (non-contiguous), transpose the tile so columns become rows
- Now vectorized row loads are contiguous and conflict-free
- This is exactly what you do in warp-tiling kernels: transpose matrix A in SMEM

**Option C: Accept scalar loads**
- If neither padding nor transposing makes sense for your access pattern, scalar loads might genuinely be better
- Modern GPUs can issue multiple scalar loads in parallel if they're conflict-free
- The instruction overhead might be less than the serialization cost

**Decision framework:**

1. **Always vectorize GMEM → SMEM**: Memory bandwidth matters more than bank conflicts here
2. **Vectorize SMEM → registers only if**:
   - Access pattern is naturally aligned (row-wise on row-major data)
   - OR you can pad/transpose to make it aligned
   - AND profiler confirms no new bank conflicts
3. **When in doubt**: Profile both ways and measure:
   - Shared memory load transactions
   - Bank conflict rate (wavefronts metric)
   - Overall kernel duration

The deeper lesson: "Best practices" exist in context. Vectorization is powerful when it aligns with your memory layout, but forcing it when the layout doesn't support it just trades one bottleneck for another. Always profile before and after.

**Q13: You're implementing warp-tiling for a GEMM kernel. You've got your block computing a 128×128 tile using 128 threads (4 warps). A colleague suggests: "Why not use 256 threads (8 warps) to get better occupancy?" You try it and performance drops by 15%. Using the roofline model and profiler metrics, explain what likely happened and why higher occupancy made things worse here.**

**A:** This is an excellent example of why occupancy is a means to an end, not the goal itself. Let's dissect what happened.

**The setup:**
- Original: 128 threads/block (4 warps), computing 128×128 output tile
- Modified: 256 threads/block (8 warps), computing same 128×128 tile
- Result: 15% performance regression despite higher occupancy

**What the profiler likely shows:**

**Original kernel (128 threads):**
- Achieved occupancy: ~50% (let's say 32 warps/SM out of 64 max)
- SM Issue Active: ~65%
- SMEM usage: 48KB per block
- Register usage: 96 registers/thread
- Blocks per SM: 2 (limited by thread count)

**Modified kernel (256 threads):**
- Achieved occupancy: ~75% (48 warps/SM)
- SM Issue Active: ~55% (worse!)
- SMEM usage: same 48KB per block
- Register usage: same 96 registers/thread
- Blocks per SM: 1 (limited by thread count: 256×1 < 2048, but you can't fit 2)

**What went wrong - multiple compounding factors:**

**1. Work distribution inefficiency**

With 128 threads computing a 128×128 tile:
- Each thread computes 128 elements (128×128 / 128 threads)
- Register tiling: each thread does 8×16 subtile
- High arithmetic intensity: lots of compute per SMEM load

With 256 threads computing the same 128×128 tile:
- Each thread computes only 64 elements (128×128 / 256 threads)
- Register tiling: each thread does 8×8 subtile
- Lower arithmetic intensity: same SMEM loads, half the compute

**2. Instruction-Level Parallelism (ILP) reduction**

Original kernel's inner loop (simplified):
```cuda
for (int k = 0; k < BK; k++) {
    for (int m = 0; m < 8; m++) {
        for (int n = 0; n < 16; n++) {
            results[m][n] += regA[m] * regB[n];
        }
    }
}
```
8×16 = 128 independent FMAs per iteration - excellent ILP!

Modified kernel:
```cuda
for (int k = 0; k < BK; k++) {
    for (int m = 0; m < 8; m++) {
        for (int n = 0; n < 8; n++) {
            results[m][n] += regA[m] * regB[n];
        }
    }
}
```
8×8 = 64 independent FMAs - half the ILP!

Lower ILP means the warp scheduler has fewer independent instructions to hide latency. The warp gets swapped out more often waiting for dependencies.

**3. Cache thrashing**

With 4 warps per block:
- Working set: 128×128 tile in SMEM (~64KB)
- L1 cache working set per SM: 2 blocks × 64KB = 128KB
- Fits reasonably in L1/SMEM combined resources

With 8 warps per block:
- Can only fit 1 block per SM (due to thread limit)
- BUT now 8 warps are thrashing the same SMEM
- More contention for SMEM banks
- Less effective L1 cache usage per warp

**4. Shared memory bank conflict pattern changes**

With 4 warps in warp-tiling:
- Each warp gets a 64×64 quadrant
- Within a warp, threads access SMEM in a predictable pattern
- With ROWS_PER_THREAD=8, COLS_PER_THREAD=16, access patterns are well-distributed

With 8 warps:
- Need to subdivide the tile differently
- Each warp might get a 64×32 slice (or similar awkward shape)
- COLS_PER_THREAD=8 can create alignment issues
- Different bank conflict characteristics

**5. Warp scheduler efficiency**

The 4-warp configuration maps perfectly to Hopper's 4 warp schedulers per SM:
- Each scheduler gets exactly 1 warp from this block
- Clean, predictable scheduling
- Minimal context switching overhead

With 8 warps from one block:
- Schedulers have to juggle 2 warps each
- More context switching
- Potential for scheduler imbalance

**Roofline model perspective:**

Let's calculate arithmetic intensity:

**Original (4 warps, 128 threads):**
- Tile size: 128×128 = 16,384 FP32 elements
- SMEM loads over K dimension: for BK=16, each 128×16 and 16×128 chunk
  - Total: 128×16 + 16×128 = 2,048 + 2,048 = 4,096 floats loaded per K-tile
  - Over full K=4096: (4096/16) × 4,096 = 1,048,576 floats = 4MB
- FLOPs: 2 × 128 × 128 × 4096 = 134,217,728
- **AI = 134M FLOPs / 4MB = ~33.5 FLOPs/byte**

**Modified (8 warps, 256 threads):**
- Same tile size: 128×128
- Same SMEM loads: 4MB (loading pattern unchanged)
- But now 256 threads means:
  - More threads doing redundant loads from SMEM (broadcast less effective)
  - Potentially more bank conflicts
  - Effective AI might drop to ~30 FLOPs/byte

Even a small drop in AI can push you from compute-bound toward memory-bound.

**The fundamental insight:**

Occupancy helps hide latency by providing alternative warps to schedule when one stalls. But:
- We're compute-bound, not latency-bound
- Adding warps dilutes the work per thread
- Lower ILP means more stalls per warp
- More warps fighting over the same resources (SMEM, L1)

The original 50% occupancy was already sufficient to hide memory latency. Going to 75% just added overhead without addressing the actual bottleneck.

**How to verify this hypothesis:**

Profile and compare:
1. `sm_efficiency` - how busy are the SMs? (Probably similar)
2. `achieved_occupancy` - confirms we increased it
3. `ipc_instance` - instructions per cycle (probably dropped)
4. `stall_memory_dependency` - data dependency stalls (probably increased)
5. `shared_load_throughput` - SMEM traffic (probably similar or higher)
6. `l1_cache_hit_rate` - L1 efficiency (probably dropped)

**When would higher occupancy help?**

If original kernel showed:
- High `stall_memory_throttle` or `stall_long_scoreboard`
- Low `sm_efficiency` due to insufficient warps to hide latency
- Memory-bound regime on roofline

But for compute-bound warp-tiling: keep threads-per-block low, maximize work per thread, prioritize ILP over TLP (thread-level parallelism).

**Q14: Walk me through the complete thought process of implementing a Hopper GEMM kernel that uses TMA + WGMMA. You're starting from a working warp-tiling baseline. What are the key design decisions you need to make, what are the tradeoffs, and what order would you implement the optimizations in? Assume you're targeting 128×256 output tiles and bf16 precision.**

**A:** This is a complex systems question that tests your ability to orchestrate multiple advanced features. Let me walk through this methodically.

**Stage 0: Baseline understanding**

We're starting from warp-tiling with:
- Block size: 128 threads (4 warps)
- Tile: 128×128, computing with FP32 CUDA cores
- Manual loading via vectorized loads (float4)
- Manual SMEM management with padding to avoid bank conflicts
- Performance: maybe 30-40 TFLOP/s

Target with Hopper features:
- TMA for async loads
- WGMMA tensor cores
- bf16 input, fp32 accumulator
- 128×256 output tile
- Target: 400+ TFLOP/s

**Stage 1: Transition to Tensor Cores (m64n64k16)**

**First decision: Block size and warp groups**

WGMMA works on warp groups (128 threads = 4 warps). This is non-negotiable.
- Keep block size at 128 threads initially
- This gives us 1 warp group for compute

**Second decision: Tile sizes**

Start conservative:
- TILE_M = 64 (matches WGMMA_M = 64)
- TILE_N = 64 (matches WGMMA_N = 64)
- TILE_K = 64 (will need 4 WGMMA instructions since WGMMA_K = 16)

Why start here?
- 1:1 mapping between output tile and single WGMMA shape
- Simplifies register layout understanding
- Easier to debug

**Third decision: Data layout**

Critical: B must be **column-major** for efficient WGMMA access!
- A: row-major (M×K)
- B: column-major (K×N) 
- C: row-major (M×N)

**Fourth decision: Accumulator organization**

Each thread in the warp group holds part of the 64×64 accumulator:
```cuda
float d[WGMMA_N/16][8];  // 64/16 = 4, so d[4][8] = 32 floats/thread
```

128 threads × 32 floats = 4096 = 64×64 ✓

**Implementation steps for Stage 1:**

1. **Convert to bf16 inputs**:
   - Change matrix types from float to bf16
   - Add conversion at boundaries if needed

2. **Implement WGMMA descriptor creation**:
   ```cuda
   __device__ uint64_t make_smem_desc(bf16* ptr) {
       uint32_t addr = __cvta_generic_to_shared(ptr);
       uint64_t desc = 0;
       desc |= encode(addr);          // bits [13:0]
       desc |= encode(16) << 16;      // leading dim offset
       desc |= encode(1024) << 32;    // stride offset (8 rows * 128B)
       desc |= 1llu << 62;            // 128B swizzle mode
       return desc;
   }
   ```

3. **Write WGMMA wrapper**:
   ```cuda
   template<int ScaleD, int ScaleA, int ScaleB, int TransA, int TransB>
   __device__ void wgmma64(float d[4][8], bf16* sA, bf16* sB) {
       uint64_t desc_a = make_smem_desc(sA);
       uint64_t desc_b = make_smem_desc(sB);
       asm volatile(
           "wgmma.mma_async.sync.aligned.m64n64k16.f32.bf16.bf16 "
           "{%0,%1,...,%31}, %32, %33, %34, %35, %36, %37, %38;\n"
           : "+f"(d[0][0]), ... "+f"(d[3][7])
           : "l"(desc_a), "l"(desc_b), ...
       );
   }
   ```

4. **Modify compute loop**:
   ```cuda
   asm volatile("wgmma.fence.sync.aligned;" ::: "memory");
   for (int k = 0; k < TILE_K/WGMMA_K; k++) {
       wgmma64<1,1,1,0,0>(d, &sA[k*WGMMA_K], &sB[k*WGMMA_K]);
   }
   asm volatile("wgmma.commit_group.sync.aligned;" ::: "memory");
   asm volatile("wgmma.wait_group.sync.aligned %0;" ::"n"(0) : "memory");
   ```

5. **Update store logic** to match WGMMA register layout (PTX docs section 9.7.15.5.1.1)

**Expected result**: ~280-320 TFLOP/s (if you do it right)

**Stage 2: Add TMA for async loads**

Now we'll replace manual loading with TMA.

**First decision: Tensor map construction**

On the host, before kernel launch:
```cuda
template<uint BM, uint BN>
CUtensorMap* create_tensor_map(bf16* ptr, uint M, uint N) {
    CUtensorMap* map;
    cudaMalloc(&map, sizeof(CUtensorMap));
    
    CUtensorMap host_map;
    uint64_t dims[5] = {N, M, 1, 1, 1};  // column-major!
    uint64_t strides[4] = {sizeof(bf16), N*sizeof(bf16), 0, 0};
    uint32_t box[5] = {BN, BM, 1, 1, 1};
    
    cuTensorMapEncodeTiled(&host_map, CU_TENSOR_MAP_DATA_TYPE_BFLOAT16,
        2, ptr, dims, strides, box, ..., 
        CU_TENSOR_MAP_SWIZZLE_128B, ...);
    
    cudaMemcpy(map, &host_map, sizeof(CUtensorMap), H2D);
    return map;
}
```

**Second decision: Barrier setup**

Need async barriers to coordinate TMA completion:
```cuda
__shared__ barrier barA, barB;

if (threadIdx.x == 0) {
    init(&barA, blockDim.x);  // 128 threads expected
    init(&barB, blockDim.x);
    fence_proxy_async_shared_cta();  // visibility fence
}
__syncthreads();  // bootstrap barrier
```

**Third decision: Loading pattern**

Only thread 0 issues TMA:
```cuda
for (int k_tile = 0; k_tile < num_k_tiles; k_tile++) {
    if (threadIdx.x == 0) {
        cp_async_bulk_tensor_2d_global_to_shared(
            &sA[0], tensorMapA, k_tile*BK, block_m*BM, barA);
        tokenA = barrier_arrive_tx(barA, 1, sizeof(sA));
        
        cp_async_bulk_tensor_2d_global_to_shared(
            &sB[0], tensorMapB, k_tile*BK, block_n*BN, barB);
        tokenB = barrier_arrive_tx(barB, 1, sizeof(sB));
    } else {
        tokenA = barA.arrive();
        tokenB = barB.arrive();
    }
    
    barA.wait(std::move(tokenA));
    barB.wait(std::move(tokenB));
    
    // WGMMA compute...
}
```

**Benefits**:
- Hardware handles swizzling automatically
- Async copy overlaps with previous iteration's compute (eventually)
- Simpler code

**Expected result**: ~300-350 TFLOP/s (modest improvement because not pipelined yet)

**Stage 3: Producer-consumer pipeline**

This is where performance really takes off.

**Key insight**: While tensor cores compute on tile K, TMA should be fetching tile K+1.

**First decision: Queue size**

Start with QSIZE = 5:
- Need enough slots to hide TMA latency
- Too many wastes SMEM
- 5 is empirically good for most cases

**Second decision: Warp group assignment**

Need 256 threads now:
- WG0 (threads 0-127): Producer (TMA coordinator)
- WG1 (threads 128-255): Consumer (WGMMA compute)

**Third decision: Barrier organization**

```cuda
__shared__ barrier full[QSIZE], empty[QSIZE];
```
- `full[i]`: signaled when slot i is ready to read
- `empty[i]`: signaled when slot i is ready to write

**Implementation**:

```cuda
int wg_idx = threadIdx.x / 128;

if (wg_idx == 0) {  // Producer
    if (threadIdx.x == 0) {
        int qidx = 0;
        for (int k = 0; k < num_k_tiles; k++) {
            empty[qidx].wait(empty[qidx].arrive());  // wait until slot free
            
            // Issue TMA
            cp_async_bulk_tensor_2d(
                &sA[qidx*BK*BM], mapA, k*BK, bm*BM, full[qidx]);
            cp_async_bulk_tensor_2d(
                &sB[qidx*BK*BN], mapB, k*BK, bn*BN, full[qidx]);
            
            barrier_arrive_tx(full[qidx], 1, sizeof(sA)+sizeof(sB));
            
            qidx = (qidx + 1) % QSIZE;
        }
    }
} else {  // Consumer
    // Mark all slots initially empty
    for (int i = 0; i < QSIZE; i++) {
        empty[i].arrive();
    }
    
    float d[WGMMA_N/16][8];
    memset(d, 0, sizeof(d));
    
    int qidx = 0;
    for (int k = 0; k < num_k_tiles; k++) {
        full[qidx].wait(full[qidx].arrive());  // wait for data
        
        // WGMMA compute on sA[qidx*...], sB[qidx*...]
        warpgroup_arrive();
        for (int ki = 0; ki < BK/WGMMA_K; ki++) {
            wgmma64(d, &sA[qidx*BK*BM + ki*WGMMA_K], 
                      &sB[qidx*BK*BN + ki*WGMMA_K]);
        }
        warpgroup_commit_batch();
        warpgroup_wait<0>();
        
        empty[qidx].arrive();  // mark slot consumed
        qidx = (qidx + 1) % QSIZE;
    }
    
    // Store results...
}
```

**Expected result**: ~420-480 TFLOP/s

**Stage 4: Grow tile to 128×256**

Now we tackle the target tile size.

**Problem**: 128×256 accumulator won't fit in one warp group's registers!
- 128×256 = 32,768 fp32 values
- Each thread has ~255 registers
- 128 threads × 255 registers = 32,640... barely fits but no room for anything else!

**Solution**: Use 2 consumer warp groups

New thread organization:
- WG0 (0-127): Producer
- WG1 (128-255): Consumer A (computes rows 0-63 of 128×256)
- WG2 (256-383): Consumer B (computes rows 64-127 of 128×256)

Now need 384 threads total.

**WGMMA shape decision**: Use m64n128k16 or m64n256k16?

Start with m64n128k16:
- Each consumer does 2 WGMMA instructions per K-step (N direction)
- 64×128 accumulator per consumer = 8,192 fp32
- 8,192/128 threads = 64 registers/thread for accumulator ✓

```cuda
float d[TILE_M/WGMMA_M][WGMMA_N/16][8];
// For TILE_M=128, WGMMA_M=64, WGMMA_N=128:
// d[2][8][8] = 128 floats/thread
```

**Compute loop per consumer**:
```cuda
warpgroup_arrive();
for (int m_it = 0; m_it < TILE_M/WGMMA_M; m_it++) {
    bf16* sA_base = sA + (wg_idx-1)*(TILE_M/2)*BK  // offset for this consumer
                      + m_it*WGMMA_M*BK;
    for (int k_it = 0; k_it < BK/WGMMA_K; k_it++) {
        wgmma<WGMMA_N=128>(d[m_it], 
            &sA_base[k_it*WGMMA_K],
            &sB[k_it*WGMMA_K]);
    }
}
warpgroup_commit_batch();
warpgroup_wait<0>();
```

**Expected result**: ~550-610 TFLOP/s

**Stage 5: Advanced optimizations**

Once the basic pipeline works:
- **Persistent kernels**: Launch one block per SM, loop over tiles to hide store latency
- **Clusters + TMA multicast**: Share loads across 2-SM clusters via DSMEM
- **Barrier optimization**: Remove redundant arrive() calls
- **Register rebalancing**: Use `setmaxnreg` to shift registers from producer to consumers
- **Async stores**: Store via SMEM+TMA to overlap write-back with next tile's compute
- **Hilbert-curve scheduling**: Better cache locality when processing tiles

**The key insight**: Each stage builds on the previous. You can't jump straight to the pipelined version—you need the working WGMMA+TMA foundation first. Debug at each stage, profile to confirm improvements, then add the next optimization.

- **Both high** → well-balanced kernel (rare on first look!)

**2. The Roofline chart**

I check where the kernel sits:
- **Below the diagonal line** → memory-bound region
- **Above the diagonal** → compute-bound region
- **How far below** → tells me how much headroom exists

The arithmetic intensity value tells me:
- **AI < 10 FLOPs/byte** → definitely memory-bound, need more reuse
- **AI ~ 30-50** → borderline, might be memory or compute
- **AI > 100** → should be compute-bound if optimized well

**3. Achieved Occupancy**

Quick sanity check:
```
Achieved Occupancy:          18.2%
Theoretical Occupancy:       50.0%
```

If achieved << theoretical, something is limiting active warps:
- Register pressure
- Shared memory usage
- Thread count mismatch
- But remember: **occupancy is not the goal!** It's just a means to hide latency

**Decision tree after 30 seconds:**

```
Memory Throughput > 75%?
├─ YES → Memory-bound path (go to Phase 2A)
└─ NO → Compute Throughput > 70%?
    ├─ YES → Compute-bound path (go to Phase 2B)  
    └─ NO → Latency/Launch issues (go to Phase 2C)
```

---

**Phase 2A: Memory-bound diagnosis**

If Memory Throughput is high (>75%), I drill into the memory hierarchy:

**Step 1: Check the Memory Workload Analysis section**

Look at the L1/TEX Cache Hit Rate:
```
L1/TEX Hit Rate:             4.3%
```

**Low hit rate (<20%)**: 
- Tells me data isn't being reused
- Every access goes to L2 or DRAM
- Classic sign: no tiling or poor tiling strategy

Then check DRAM vs L2:
```
DRAM Throughput:             623 GB/s  (85% of peak)
L2 Throughput:               289 GB/s  (12% of peak)
```

**High DRAM, low L2**:
- Poor L2 cache utilization
- Data not reused across blocks
- Possible solution: better block scheduling, persistent kernels

**Step 2: Check global memory efficiency**

```
Global Load Efficiency:      32.1%
Global Store Efficiency:     98.7%
```

**Low load efficiency (<80%)**:
- Uncoalesced accesses! Memory transactions wasted
- Check access pattern: are consecutive threads reading consecutive addresses?
- Solution: fix indexing, transpose data layout, or use shared memory staging

**Step 3: Check shared memory bank conflicts**

```
Shared Memory Bank Conflicts: 4.8 (avg way)
```

**Conflicts > 2-way**:
- Threads within warp accessing same bank, different addresses
- Serializes access (4.8× slowdown here!)
- Solution: padding, swizzling, or transpose layout

**My mental checklist for memory-bound kernels:**
1. ✓ Coalescing: Is global_load_efficiency > 80%?
2. ✓ Tiling: Is L1 hit rate > 40% for reused data?
3. ✓ Bank conflicts: Are shared loads/stores conflict-free?
4. ✓ Arithmetic intensity: Can I compute more per byte loaded?
5. ✓ Vectorization: Am I using float4/LDG.128 instructions?

---

**Phase 2B: Compute-bound diagnosis**

If Compute Throughput is high (>70%), I check if we're actually utilizing the right units:

**Step 1: Check the SM instruction breakdown**

```
SM:
  Pipe FMA Cycles Active:        62.3%
  Pipe FP16 Cycles Active:       0.0%
  Pipe Tensor Cycles Active:     0.0%
  Pipe ALU Cycles Active:        12.4%
  Pipe Lsu Cycles Active:        18.2%
```

This is CRITICAL. The "Compute Throughput" headline might say 70%, but **which** compute units?

**High Pipe LSU, low Pipe FMA**:
- Despite "compute-bound" label, we're actually doing loads/stores!
- The overview is misleading (LSU counts as "SM activity")
- Real problem: too many memory instructions, not enough math
- Solution: increase arithmetic intensity via register tiling

**Only FP32 FMA active, but we have fp16 data**:
- Not using tensor cores!
- Leaving massive performance on the table
- fp16 tensor cores: ~989 TFLOP/s vs FP32 CUDA cores: ~67 TFLOP/s
- Solution: switch to WMMA/MMA/WGMMA

**Step 2: Check IPC (Instructions Per Cycle)**

```
IPC:                              1.2
```

**IPC < 2.0 for compute-bound**:
- Not enough instruction-level parallelism (ILP)
- Warps stalling on dependencies
- Solution: loop unrolling, independent operations

**Step 3: Check warp stall reasons**

```
Warp Stall Reasons:
  Stall Short Scoreboard:        28.3%
  Stall Long Scoreboard:         15.2%
  Stall Wait:                    12.1%
```

**High scoreboard stalls**:
- Waiting for previous instruction results
- Low ILP within each warp
- Solution: unroll more, introduce independent operations

**My mental checklist for compute-bound kernels:**
1. ✓ Right hardware: Tensor cores for matmul? SFU for transcendentals?
2. ✓ ILP: Is loop unrolled enough? Can threads do independent work?
3. ✓ Precision: Can I use fp16/bf16 instead of fp32?
4. ✓ Instruction mix: Am I doing real compute or hidden memory ops?

---

**Phase 2C: Latency-bound or other issues**

If both compute and memory throughput are low (<50%), I look for:

**Step 1: Check SM efficiency**

```
SM Efficiency:                   42.1%
```

**Low SM efficiency (<60%)**:
- SMs sitting idle
- Not enough blocks to saturate GPU
- Check: how many blocks launched? GPU has 114 SMs, need many more blocks

**Step 2: Check wave quantization**

```
Waves Per SM:                    0.8
```

**Waves < 1.0**:
- Launched fewer blocks than SMs!
- Terrible GPU utilization
- Solution: increase grid size, reduce block size

**Step 3: Check launch overhead**

If kernel is very short (<100μs):
```
Kernel Duration:                 23μs
```

Launch overhead might dominate:
- CPU→GPU launch: ~5-10μs baseline
- Grid setup, parameter passing overhead
- Solution: fuse with other kernels, use CUDA graphs

**Step 4: Check occupancy limiters**

```
Occupancy Limited By:            Registers (96/thread)
Registers Per Thread:            96
Shared Memory Per Block:         48 KB
Threads Per Block:               256
```

**Occupancy limited by registers**:
- Too many registers/thread, can't fit enough warps
- May or may not matter! Check if we have stalls due to insufficient warps
- If `Stall MIO Throttle` or `Stall Long Scoreboard` are high: need more warps
- Solution: reduce register usage, increase threads/block

---

**Phase 3: Code inspection (only after profiler)**

Now I look at the actual code with profiler insights in mind:

**If memory-bound:**
1. Check indexing patterns: `A[row * N + col]` — is this coalesced?
2. Check for shared memory: Is there tiling? Is data loaded once and reused?
3. Check tile sizes: Are they too small (waste parallelism) or too large (waste cache)?

**If compute-bound (but wrong units):**
1. Check data types: `float*` where `half*` or `__nv_bfloat16*` would work?
2. Check for explicit tensor core usage: any `wmma`, `mma.sync`, `wgmma`?
3. Check loop structure: Is there enough unrolling for ILP?

**If latency-bound:**
1. Check launch config: `<<<grid, block>>>` — are there enough blocks?
2. Check for unnecessary sync: too many `__syncthreads()` calls?
3. Check for resource limits: per-thread arrays that are too large?

---

**My systematic approach summary:**

```
1. Profile with ncu --set full                    [30 seconds]
2. SOL section → Memory or Compute bound?         [30 seconds]
3. If Memory → check efficiency, conflicts, L1    [2 minutes]
4. If Compute → check which pipes, IPC, stalls    [2 minutes]  
5. If Both low → check SM util, waves, occupancy  [2 minutes]
6. Form hypothesis: "bottleneck is X because Y"   [1 minute]
7. Code inspection to verify hypothesis           [5 minutes]
8. Make targeted fix                              [varies]
9. Re-profile to confirm                          [1 minute]
```

**Total time to diagnosis: ~10 minutes maximum**

**The golden rules:**

1. **Profiler first, code second** — never guess based on code alone
2. **One hypothesis at a time** — don't change multiple things
3. **Always re-profile** — verify your fix actually helped
4. **The bottleneck shifts** — fixing memory issues often exposes compute issues
5. **Occupancy is not the goal** — it's a tool to hide latency, not an end

**Common traps to avoid:**

- "It says 75% compute throughput so it's compute-bound" → Check WHICH pipes!
- "Occupancy is only 25%, must fix that first" → Maybe you're compute-bound with high ILP
- "I'll add __launch_bounds__ to increase occupancy" → Might make things worse
- "Let me add shared memory" → If you're compute-bound, this won't help

The key is: **let the profiler tell you what's wrong, don't assume**. Every kernel is different, and intuition fails more often than it succeeds.

## Core Performance Analysis Questions

**Q1: You've profiled your custom attention kernel and notice it's achieving only 40% of theoretical memory bandwidth. Your kernel is memory-bound (arithmetic intensity ~0.5 FLOPs/byte). Walk me through your debugging process. What metrics would you examine first, and what are the most common culprits that could cause this gap?**

**A:** I'd approach this systematically:

First, I'd check **memory transaction efficiency metrics** in nsight compute:
- `l2_tex_read_throughput` vs theoretical bandwidth
- `global_load_efficiency` and `global_store_efficiency` - if these are low (<80%), we have wasted transactions
- Sector utilization - are we requesting 32 bytes but only using a few?

Common culprits in order of likelihood:
1. **Uncoalesced memory access** - If consecutive threads in a warp aren't accessing consecutive memory locations, each thread might trigger separate 32-byte transactions. For attention, this often happens with non-contiguous head access patterns.
2. **Bank conflicts in shared memory** - If using shared memory as an intermediary, broadcast or multicast bank conflicts could serialize access and create bottlenecks
3. **Cache thrashing** - L2 cache misses due to poor data reuse. Check `l2_tex_hit_rate`
4. **Memory-level parallelism** - Not enough concurrent memory requests in flight. Check `sm_efficiency` - if SMs are idle waiting for memory, we need more independent memory operations per thread or more warps
5. **TLB misses** - Less common but can happen with very large tensors and scattered access patterns

For attention specifically, I'd verify the Q, K, V tensor layouts are optimized for the access pattern - often transposing K or reordering dimensions can dramatically improve coalescing.

**Q2: You're implementing a fused kernel that combines layer normalization and GELU activation. The intermediate statistics (mean, variance) need to be computed across the hidden dimension (say 4096 elements). You have two architectural choices: (a) use one warp per row with warp-level reductions, or (b) use multiple warps per row with block-level reductions. How would you decide between these approaches, and what are the tradeoffs?**

**A:** This is a fascinating tradeoff between parallelism granularity and reduction overhead.

**Single warp per row (a):**
- Pros:
  - No synchronization overhead beyond implicit warp-synchronous execution
  - No shared memory contention or atomics needed
  - Better for smaller hidden dimensions (512-2048)
  - Simpler code, fewer edge cases
- Cons:
  - Limited parallelism - only 32 threads working on 4096 elements means each thread processes ~128 elements
  - For very large hidden dims, each thread does a lot of work serially, reducing GPU utilization
  - Wave quantization issues - if batch_size * seq_len isn't a multiple of warp count, we waste SM resources

**Multiple warps per row (b):**
- Pros:
  - Higher parallelism - can keep more threads busy
  - Better for large hidden dimensions (4096+)
  - Each thread does less serial work
  - Can achieve better memory bandwidth utilization with more concurrent requests
- Cons:
  - Need block-level reduction with `__syncthreads()` barriers
  - Shared memory usage for partial sums
  - More complex warp cooperation logic

**Decision framework:**
1. **Hidden dimension size**: <2048 → single warp; ≥4096 → multiple warps
2. **Batch × sequence length**: If very large, multiple warps help saturate the GPU
3. **Occupancy**: Check if single-warp approach leaves SMs underutilized
4. **Profile**: Measure `achieved_occupancy` and `sm_efficiency`

For 4096 hidden dim, I'd likely use 4 warps per row (128 threads), giving each thread 32 elements to process. This balances parallelism with reduction overhead. I'd verify with profiling that the `__syncthreads()` cost is amortized by better memory throughput.

**Q3: Your SGEMM kernel runs at 85% of cuBLAS performance for square matrices (M=N=K=4096), but drops to 45% for skinny matrices (M=8192, N=64, K=4096). The skinny case is critical for your ML workload. What's likely happening and how would you optimize for this case?**

**A:** This is a classic case of poor work distribution and tile size mismatch.

**Root cause analysis:**

For square matrices, we have a nice 2D grid of blocks computing output tiles. With M=N=4096 and say 128×128 tiles, we get 32×32=1024 blocks - great parallelism.

For skinny matrices (M=8192, N=64):
- If using the same 128×128 tile, N=64 means we can't even fill one tile width
- We might only launch 8192/128 × 64/128 = 64 × 1 blocks = 64 blocks total
- This massively underutilizes the GPU (modern GPUs have 80-132 SMs)
- **Wave quantization loss** - with only 64 blocks, many SMs sit idle

Additionally:
- The K=4096 dimension means each block still does the same amount of work loading A and B tiles
- But we're computing far fewer output elements per tile
- Arithmetic intensity drops because we're loading the same amount of data but doing fewer computations

**Optimization strategies:**

1. **Adaptive tile sizing**: Use rectangular tiles optimized for skinny shapes. For N=64, use something like 128×32 or 128×64 tiles to launch more blocks in the M dimension
   
2. **Increase M-dimension parallelism**: Split M into smaller chunks, launching 8192/64 = 128 blocks in M direction with 2-4 blocks in N direction

3. **Vectorized loads for the N dimension**: Since N is small, we can load entire rows with vector instructions (float4) and keep them in registers

4. **Different kernel variant**: Consider a GEMV-like approach where each warp computes multiple rows of output, treating it as multiple vector-matrix multiplies

5. **Warp specialization**: Have some warps focus on loading A tiles, others on computing, since the N dimension doesn't provide enough parallelism for standard tiling

The key insight is that one-size-fits-all tiling fails for extreme aspect ratios. I'd implement a heuristic that selects tile dimensions based on M, N, K and dispatches to specialized kernels for skinny vs square vs tall matrices.

## Occupancy & Resource Management

**Q4: You launch a kernel with 256 threads per block and it reports 25% theoretical occupancy. You reduce to 128 threads per block and occupancy increases to 50%, but your kernel actually runs slower. The profiler shows "Warp was stalled waiting for memory" as the top stall reason in both cases. Explain this apparent contradiction - why did higher occupancy make things worse?**

**A:** This is a great example of why occupancy is a means to an end, not the goal itself.

**What's happening:**

Occupancy measures what percentage of maximum resident warps are actually resident. Higher occupancy means more warps can be scheduled to hide latency. However, there are several reasons why 50% occupancy ran slower:

1. **Reduced work per SM**: With 256 threads, fewer blocks fit per SM due to register or shared memory limits. With 128 threads, we can fit more blocks, but if our grid size is fixed, we might be doing the same total work with more blocks - this adds block-level overhead (block initialization, warp scheduling overhead)

2. **Cache thrashing**: More concurrent warps means more unique memory access patterns competing for L1/L2 cache. If our working set barely fit in cache with 4 warps (256 threads), doubling to 8 warps (if we fit 4 blocks of 128 instead of 2 blocks of 256) could evict useful data, increasing memory latency

3. **Memory bandwidth saturation**: We're already memory-bound (stalled on memory). More warps issuing memory requests doesn't help if we've already saturated DRAM bandwidth. In fact, more requests can increase queueing latency

4. **Shared memory pressure**: If using shared memory, halving block size might mean less data reuse per block, requiring more global memory round-trips

5. **ILP vs TLP tradeoff**: With 256 threads, the compiler might have achieved better instruction-level parallelism (ILP) per thread with more registers. With 128 threads and more register pressure from higher occupancy, ILP might decrease

**The key insight**: Occupancy helps hide latency through thread-level parallelism (TLP), but only if you're latency-bound AND not saturating bandwidth. Here we're bandwidth-bound, so more threads just create more contention.

**What to try instead:**
- Improve memory access patterns (coalescing, tiling)
- Increase arithmetic intensity by doing more work per loaded byte
- Use shared memory to increase data reuse
- Check if we can use async copies or pipeline stages to better overlap compute and memory

**Q5: You're tuning a flash attention kernel and notice that using 64KB of shared memory per block with 4 warps gives better performance than 32KB with 8 warps, even though the latter achieves higher occupancy (50% vs 25%). Your GPU has 100KB shared memory per SM. Explain what factors could make the lower-occupancy configuration faster, and how you'd validate your hypothesis.**

**A:** Flash attention is a perfect case study for the complex tradeoffs between occupancy and shared memory utilization.

**Factors favoring 64KB/4 warps:**

1. **Larger tile sizes, fewer reloads**: With 64KB, you can load bigger blocks of Q, K, V matrices. For example, a 64×64 Q tile + 64×128 K tile in FP16 (2 bytes each) uses ~32KB. Larger tiles mean:
   - Fewer iterations in the outer loop over K,V sequence length
   - More reuse of Q within each block before reloading
   - Amortized softmax computation over larger blocks

2. **Reduced softmax rescaling overhead**: Flash attention requires tracking running max and sum for numerical stability. Larger tiles mean fewer rescaling operations since you process more of the K,V sequence per iteration

3. **Better data reuse**: Even though we have fewer warps to hide latency, the larger tile size means each byte loaded from HBM is reused more times in shared memory. This increases arithmetic intensity.

4. **Fewer block launches**: If sequence length is 2048 and we process 128 K,V elements per block vs 64, that's 16 iterations vs 32 iterations - less loop overhead and synchronization

5. **L2 cache benefits**: Fewer blocks resident per SM can actually help L2 cache hit rates. With 4 blocks on 4 SMs, there's less cache pollution than 8 blocks on 2 SMs

6. **Warp scheduler efficiency**: 4 warps might be the sweet spot where the scheduler can efficiently interleave them without excessive context switching overhead

**Validation approach:**

```
Metrics to compare (NSight Compute):

1. Global memory throughput:
   - Check if 64KB case has fewer HBM transactions
   - Compare `dram_read_bytes` - should be lower for 64KB

2. Shared memory efficiency:
   - `shared_load_transactions` and `shared_store_transactions`
   - Bank conflict rate - 64KB might have better aligned access

3. Arithmetic intensity:
   - FLOPs / bytes_from_DRAM - should be higher for 64KB

4. Warp stall reasons:
   - If 8-warp case shows "stall_memory_throttle", we're saturating bandwidth
   - If 4-warp case shows "stall_wait", we might benefit from more occupancy (but we don't)

5. SM utilization:
   - Check if 64KB case keeps SMs busy despite lower occupancy
   - Look at `sm_active_cycles_pct`

6. Softmax rescale count:
   - Instrument code to count rescaling operations
   - Should be ~2x more for 32KB case
```

The validation would likely show that the 64KB configuration does fewer global memory transactions and achieves higher arithmetic intensity despite lower occupancy - a classic example where memory hierarchy efficiency trumps raw parallelism.

## Performance Analysis & Profiling

**Q6: You've written a custom triton kernel for fused attention that achieves 90% of Flash Attention v2 performance. Marketing wants to know if you can beat Flash Attention. You profile both kernels and see yours achieves 75% of peak FLOPS while Flash Attention achieves 65%, yet Flash Attention is faster overall. What's going on, and what metrics would you examine to understand the full picture?**

**A:** This is a masterclass in why FLOP utilization alone is misleading, especially for memory-bound operations.

**What's likely happening:**

Flash Attention is doing **fewer total FLOPs** through algorithmic efficiency. The key insight of Flash Attention is avoiding materialization of the full N×N attention matrix. Even though their FLOP utilization is lower, they're doing less total work.

**Critical metrics to examine:**

1. **Total FLOP count** (not just utilization):
   - Your kernel: `total_flops = flops_per_second × kernel_duration`
   - Flash Attention might have 30-40% fewer total FLOPs through tiling and online softmax

2. **Memory traffic**:
   - `dram_read_bytes + dram_write_bytes`
   - Flash Attention's brilliance is reducing HBM accesses from O(N²) to O(N)
   - You might be reading/writing intermediate attention scores to HBM

3. **Arithmetic intensity**:
   - Your kernel might be doing more FLOPs per byte, which explains higher FLOP utilization
   - But if you're loading more bytes overall, the total time is longer

4. **Kernel duration breakdown**:
   ```
   Your kernel: 2.0ms (75% FLOP util)
   Flash Attention: 1.8ms (65% FLOP util)
   
   But:
   Your kernel: 8 TFLOPs total work
   Flash Attention: 5 TFLOPs total work
   
   8 TFLOPs / 2.0ms = 4 PFLOPS/s = 75% of 5.33 peak
   5 TFLOPs / 1.8ms = 2.78 PFLOPS/s = 65% of 5.33 peak
   ```

5. **Memory efficiency metrics**:
   - `l2_atomic_throughput` - are you doing atomics that Flash Attention avoids?
   - `global_hit_rate` - cache efficiency
   - Number of shared memory transactions

6. **Algorithmic differences**:
   - Count softmax operations - are you computing softmax over full sequence each time?
   - Inspect if you're doing redundant computation in overlapping tiles

**The lesson**: 

High FLOP utilization often means "I'm doing a lot of math per byte loaded," but if you're loading too many bytes or doing unnecessary math, you'll lose. Flash Attention wins by fundamentally reducing the O(N² d) memory footprint to O(N d) through clever tiling and online statistics.

To beat Flash Attention, you'd need to either:
- Match their algorithmic efficiency (hard - it's patented and highly optimized)
- Find a different algorithmic approach that does even less work
- Exploit specific hardware features they don't (tensor cores, async copies, etc.)

The answer to marketing: "We achieve higher computational efficiency, but Flash Attention has fundamental algorithmic advantages that reduce total work. To beat them, we'd need algorithmic innovation, not just better CUDA code."

**Q7: You're optimizing a beam search kernel for inference. You notice that for batch_size=1, vocab_size=50K, beam_width=4, your kernel takes 0.5ms. When you increase batch_size=32, it takes 8ms instead of the expected ~16ms. You're happy about the superlinear speedup, but your manager asks you to explain it. What's happening and why?**

**A:** This is a beautiful example of how GPU efficiency scales with problem size, and understanding this is crucial for inference optimization.

**Why superlinear speedup happens:**

1. **Fixed launch overhead amortization**:
   - Kernel launch has fixed overhead (~1-10μs) regardless of work
   - For batch=1, maybe 50μs of overhead / 500μs total = 10% waste
   - For batch=32, same 50μs / 8000μs = 0.6% waste
   - But this alone doesn't explain 16ms → 8ms

2. **Improved occupancy and wave quantization**:
   - For batch=1, beam=4, you might launch only 4 blocks or a small grid
   - If GPU has 80 SMs and you launch 4 blocks, 76 SMs sit idle
   - For batch=32, beam=4, you launch 128 blocks → better SM utilization
   - **Wave quantization**: blocks execute in waves. If you have 76 SMs and 4 blocks, you severely underutilize

3. **Better memory coalescing**:
   - With batch=1, consecutive threads might be processing consecutive vocabulary items but there's limited inter-thread data reuse
   - With batch=32, you can organize threads so consecutive threads handle consecutive batch items for the same vocab item
   - This improves coalescing: 32 threads in a warp each reading from different batch items but same vocab position = one coalesced transaction

4. **Shared memory utilization**:
   - Beam search typically needs top-k selection, which benefits from block-level reductions
   - For batch=1, you might use 10KB of 100KB available shared memory per block
   - For batch=32, each SM can fit fewer blocks but each block is doing more useful work with the same shared memory
   - Better amortization of shared memory latency

5. **Cache efficiency**:
   - Vocabulary embeddings (50K × hidden_dim) might not fit in L2 cache
   - With batch=1, each vocabulary item is loaded once, little reuse
   - With batch=32, the same vocab items are accessed across multiple batch items
   - If processing batch items in the right order, you get L2 cache hits
   - For example, if all 32 batches examine vocab item 0 before moving to vocab item 1, better cache reuse

6. **Reduced memory bandwidth bottleneck**:
   - For batch=1, you might have low arithmetic intensity - lot of memory loads per computation
   - For batch=32, same logits are compared across beams for multiple batch items
   - More computation per byte loaded from HBM

**Validation**:

Check these metrics for batch=1 vs batch=32:
- `achieved_occupancy`: expect much higher for batch=32
- `sm_efficiency`: should improve dramatically
- `l2_cache_hit_rate`: higher for batch=32
- `dram_read_throughput`: might be similar, but total bytes might not scale linearly
- `warp_execution_efficiency`: should improve

**Implications for deployment**:

This suggests batching multiple requests together whenever possible during inference. Even with increased latency per request (8ms/32 = 0.25ms per item vs 0.5ms for solo batch), total throughput is much better (32/8ms = 4 items/ms vs 1/0.5ms = 2 items/ms).

## Kernel Design & Optimization

**Q8: You're implementing a fused RMSNorm + Residual + Dropout kernel. The naive approach computes RMSNorm statistics (sum of squares), normalizes, adds residual, and applies dropout in sequence. A colleague suggests computing everything in a single pass over the data. What are the tradeoffs, and under what conditions would you choose each approach?**

**A:** This is a deep question about memory access patterns, data dependencies, and the fundamental tradeoff between parallelism and memory efficiency.

**Single-pass approach:**

Pros:
- Only one read of input tensor from HBM
- Only one write of output to HBM
- Better cache utilization if working set fits in cache
- Minimal global memory bandwidth: 2× hidden_dim × batch × seq (one read, one write)

Cons:
- **Requires two-pass algorithm for RMSNorm** or specialized reduction
  - First pass: compute sum of squares across hidden dimension
  - But in "single pass" over data, we need partial results
- **Complex implementation**: need to coordinate between normalization statistics computation and element-wise operations
- **Potential for lower parallelism**: if we need a warp or block to finish reduction before proceeding
- **Register pressure**: holding residual values, normalization state, dropout masks simultaneously

**Multi-pass approach:**

Pros:
- **Simpler**: each operation is independent, easier to verify correctness
- **More parallelism**: separate kernels can have optimal grid/block dimensions for each operation
- **Easier to optimize individually**: can tune each kernel separately
- **Less register pressure**: each kernel uses registers for its specific task

Cons:
- **Multiple HBM round-trips**: 
  - Read input for RMSNorm, write normalized output
  - Read normalized + read residual, write sum
  - Read sum, write dropout output
  - ~4-5 HBM reads, 3 writes
- **Kernel launch overhead**: 3-4 separate kernel launches (microseconds each, but adds up)
- **Less cache reuse**: data might be evicted between kernels

**When to choose each:**

**Choose single-pass if:**
1. **Hidden dimension is small** (≤2048): Reduction fits nicely in a warp or small block, easy to manage
2. **Bandwidth-bound regime**: If you're saturating HBM bandwidth, reducing memory traffic by 3× is huge
3. **High batch × seq_len**: Enough parallelism across the batch/sequence dimension that we don't sacrifice occupancy
4. **Latency-critical**: Reducing kernel launches helps minimize latency

**Choose multi-pass if:**
1. **Hidden dimension is large** (≥4096): Reduction is complex, might as well do it optimally in a separate kernel
2. **Compute-bound**: If operations are heavy (dropout with complex masks, non-standard residual), kernel launch overhead is negligible
3. **Debugging/iteration**: Easier to verify correctness and tune each stage
4. **Code maintainability**: Simpler code that future engineers can understand

**Hybrid approach** (my preference for production):

Use a two-kernel approach:
1. **Kernel 1**: Fused RMSNorm + Residual (read input, read residual, write normalized+residual sum)
2. **Kernel 2**: Dropout (read from kernel 1 output, write final result)

This achieves:
- Only 3 HBM accesses instead of 5-7 (single read of original input, single read of residual, one intermediate write, final write)
- Moderate complexity
- Good cache locality for the heavy lifting (RMSNorm)
- Simpler dropout kernel that's easy to verify

For hidden_dim ≤ 2048, I'd further fuse into a single kernel using warp-level reductions for the RMS statistic.

**Q9: You're writing a batched matrix multiplication kernel where matrices have varying sizes (M, N, K all different per batch item). One approach is to launch a single kernel with a 3D grid (batch, M-tiles, N-tiles) and use dynamic indexing. Another is to sort batches by size, group similar sizes, and launch multiple kernels. What factors determine which approach is better, and how would you implement the decision heuristic?**

**A:** This is a classic systems problem that appears in production ML inference: handling heterogeneous workloads efficiently.

**Single kernel with 3D grid approach:**

Pros:
- Single kernel launch (minimal overhead)
- Simpler memory management - one allocation for all outputs
- Easier implementation - no sorting/grouping logic

Cons:
- **Severe work imbalance**: Blocks processing small matrices finish early and idle while blocks on large matrices continue
- **Poor resource utilization**: Thread blocks are sized for the largest matrix, wasting threads for smaller ones
- **Suboptimal tile sizes**: A tile size optimal for 1024×1024 is terrible for 64×64
- **Wave quantization at batch level**: If batch item 0 is huge and launches 1000 blocks, while item 1 is tiny and launches 4 blocks, massive imbalance
- **Wasted shared memory**: Allocating 64KB for all blocks when small matrices need only 16KB means fewer concurrent blocks

**Multi-kernel with grouping:**

Pros:
- **Optimal kernel config per size**: 128×128 tiles for large, 32×32 for small
- **Better load balancing**: All blocks in a kernel do similar work
- **Higher occupancy**: Small matrix kernels use less shared memory, allowing more blocks per SM
- **Can use specialized kernels**: Small matrices might benefit from different algorithms (even a GEMV-style approach for very skinny)

Cons:
- **Sorting overhead**: Need to group batches by size
- **Multiple kernel launches**: If you have 10 size groups, that's 10 launches (though this is often negligible compared to computation)
- **Memory scatter**: Outputs for sorted batches need to be reordered back to original positions
- **Complex dispatch logic**: Need runtime heuristics to decide grouping

**Decision heuristic:**

I'd implement a hybrid approach with this logic:

```cpp
// Pseudo-code for dispatch decision

struct MatrixSize { int M, N, K; };
struct BatchItem { MatrixSize size; void* A; void* B; void* C; };

void batchedGEMM(BatchItem* items, int batch_size) {
    // 1. Collect statistics
    int max_flops = 0, min_flops = INT_MAX, total_flops = 0;
    for (int i = 0; i < batch_size; i++) {
        int flops = 2 * M * N * K;
        max_flops = max(max_flops, flops);
        min_flops = min(min_flops, flops);
        total_flops += flops;
    }
    
    float imbalance = (float)max_flops / (total_flops / batch_size);
    
    // 2. Decision logic
    if (batch_size < 8) {
        // Too few items to benefit from grouping
        launchSingleKernel3D(items, batch_size);
    }
    else if (imbalance < 2.0) {
        // Sizes are relatively balanced
        launchSingleKernel3D(items, batch_size);
    }
    else if (max_flops / min_flops > 10) {
        // Huge variance - definitely worth grouping
        auto groups = groupBySimilarSize(items, batch_size);
        for (auto& group : groups) {
            launchOptimizedKernel(group);
        }
    }
    else {
        // Moderate variance - use a hybrid
        // Separate "small" vs "large" in 2-3 groups
        auto groups = groupIntoTiers(items, batch_size, num_tiers=3);
        for (auto& group : groups) {
            launchOptimizedKernel(group);
        }
    }
}
```

**Key factors:**

1. **Batch size**: <8 items → single kernel. >32 items → worth sorting
2. **Variance ratio** (max_work / min_work): >10× → definitely group
3. **Absolute sizes**: If all matrices are small (<256×256), even variance doesn't matter much, single kernel is fine
4. **Kernel launch overhead tolerance**: For inference latency, minimize launches; for throughput, optimize work distribution

**Practical implementation notes:**

- Use a **binning approach**: group matrices into size buckets (e.g., 0-256, 256-512, 512-1024, 1024+)
- **Cache kernel configurations**: pre-compile kernels for common size ranges
- **Profile-guided**: in production, track size distributions and optimize for the common case
- Consider **stream-based concurrency**: launch different size groups on different CUDA streams to overlap

For a production ML inference system, I'd likely use 3-4 size tiers with pre-tuned kernels for each, as the sorting overhead (~microseconds) is trivial compared to the 10-30% performance gain from better work distribution.

**Q10: Your model uses a custom low-bit quantized GEMM kernel (4-bit weights, 16-bit activations). You notice that on A100 (Ampere) it runs at 70% efficiency, but on H100 (Hopper) it runs at only 40% efficiency. You expected H100 to be faster. What could explain this, and how would you investigate?**

**A:** This is a fascinating architecture-specific optimization problem that highlights how kernel design must evolve with hardware.

**Potential explanations:**

1. **Tensor Core utilization mismatch**:
   - **Ampere tensor cores**: Support FP16, BF16, TF32, INT8, INT4 natively
   - **Hopper tensor cores**: Significantly more powerful but might have different optimal tile sizes
   - Your kernel might be using tile sizes optimal for A100 (e.g., 16×16×16 for mma.m16n8k16) but H100 might prefer larger tiles (64×64×64 for wgmma)
   - Hopper's new **wgmma** (warp-group matrix multiply) instructions work on entire warp groups (128 threads), not single warps (32 threads)
   - If your kernel still uses old mma instructions, you're not utilizing Hopper's architecture

2. **Memory hierarchy changes**:
   - H100 has larger L2 cache (50MB vs 40MB) but different access patterns
   - H100's HBM3 has higher bandwidth but also higher latency
   - Your kernel might be optimized for A100's latency/bandwidth ratio
   - If you're not generating enough memory-level parallelism, H100's higher latency hurts

3. **Async operations not utilized**:
   - Hopper introduced **asynchronous tensor memory accelerator (TMA)** for async copy
   - If you're using old-style `__ldgsts` (load global store shared) instead of TMA, you're wasting Hopper's capabilities
   - TMA can reduce shared memory traffic latency significantly
   - Your A100-optimized kernel likely doesn't use TMA, missing 20-30% potential speedup

4. **Thread block cluster feature unused**:
   - Hopper introduced **thread block clusters** allowing synchronization across multiple blocks
   - This enables shared memory sharing between blocks in a cluster
   - For quantized GEMM, you could potentially dequantize weights once and share across cluster blocks
   - Not using this means redundant dequantization work

5. **Different occupancy sweet spots**:
   - Hopper has 128KB shared memory per SM (vs 164KB on A100 but different partitioning)
   - Your kernel might be register-bound on H100 due to different register file organization
   - Check `achieved_occupancy` on both architectures

6. **Dequantization overhead**:
   - 4-bit → 16-bit dequantization might be done inefficiently
   - Hopper could benefit from vectorized dequantization using new instructions
   - If using bit manipulation, latency characteristics differ between architectures

**Investigation approach:**

```
Step 1: Profile on both GPUs with NSight Compute

A100 metrics to capture:
- sm_efficiency, achieved_occupancy
- tensor_precision_fu_utilization (tensor core usage)
- l2_cache_hit_rate
- dram_throughput
- stall_mio_throttle (memory instruction queue)

H100 metrics to capture:
- Same as above, plus:
- tma_transactions (if available)
- wgmma instruction count
- cluster_sync overhead

Step 2: Check architectural usage
- Are you using mma.sync or wgmma?
- Are you using TMA or old load/store?
- Tile sizes: A100 optimal vs H100 optimal
- Check SASS disassembly for instruction mix

Step 3: Microbenchmarks
- Isolate dequantization: benchmark 4bit→16bit conversion alone
- Benchmark mma vs wgmma with same tile size
- Test async copy vs sync copy

Step 4: Roofline analysis
- Plot arithmetic intensity vs achieved performance
- Check if you're hitting compute bound or memory bound
- H100 has 2× compute vs A100 but only 1.5× memory bandwidth
  - If memory bound, you won't see 2× speedup
  - At 40% efficiency, likely memory bound on H100
```

**Most likely culprit**: 

You're using Ampere-era mma instructions with 16×8×16 tiles, while H100's wgmma with 64×64×64
